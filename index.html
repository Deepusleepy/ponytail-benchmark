Ponytail Benchmark

Ponytail is a skill that makes an AI write less code. We measured whether that is a good trade, over 480 real builds on Opus 4.8.

The short answer
~44%
less code, and it still works, and blocks the attacks we ran.

The bottom line, in four numbers

Three are wins and one is a watch-out. The rest of the page shows how we got these numbers.

What we actually tested

24 real coding jobs, built four ways, five times each.

We gave a model 24 normal coding jobs. We built each one four ways: once with no plugin, then with the ponytail plugin at each of its three levels, Lite, Full, and Ultra (Full is the plugin's default). We did that five times for each, then graded every result by running the code, not by reading it.

The three plugin levels

Ponytail is a dial. Turn it up for leaner code.

Turning it up keeps shrinking the code, steadily. The catch shows up only on jobs with hidden edge cases: there, every level strips the guards and the code breaks on bad input, and Ultra was the most fragile of all. So if your job has unstated edge cases, no level is really safe and Ultra is the riskiest. If it does not, Ultra is close to a free win.

Two kinds of less

Writing less is not the same as less code.

For each job we measured two things: how much less the model wrote (output), and how much smaller the actual parsed code got. Usually they move together, but not always. Turning it up

Higher levels keep shrinking code. On fragile jobs they do not get safer.

Two different stories. The code keeps shrinking as you turn the plugin up, clean and steady. But on the jobs with hidden edge cases, the leaner code breaks at every level, and Ultra was the most fragile of all. There is no setting that buys you the savings without the risk on that kind of job. (Averaged over all 24 jobs the bad-input pass rate looks milder, from 97% to about 83%, but that average is misleading: it blends the 19 jobs that never broke with the few that broke badly.)

How we tested, and why we trust it

For the curious: the method, why our first try was no good, and what this still cannot tell you.

Why this benchmark exists

A test the tool cannot fail tells you nothing.

Earlier tests of ponytail (the author's, and our own v1) mostly asked "does it write less code?" and got a yes. But to know if less code is worse, the test has to be able to catch it failing. So we built jobs ponytail could actually trip on: jobs that tempt a model to over-build, jobs with edge cases we left unsaid on purpose, and real security holes to defend against. Then we graded by running the code and firing real attacks at it.

Why our v1 was not trustworthy
  • The skill was pasted into the prompt by hand, not run as the real installed plugin. That is not how anyone uses it, and it can flatter the result.
  • "Safety" was judged by another AI reading the code and guessing. Reading code is not the same as attacking it, so a "secure" verdict was an opinion, not a test.
How v3 fixed it
  • Run the real plugin the way you actually would, through the command-line tool, with the skill switched on per run and verified.
  • Grade by running the code: correctness by executing it against hidden tests, security by firing real attacks (path traversal, SQL injection, forged tokens) and checking what leaks.
  • Harder jobs on purpose, so a normal model fails some of them, which is the only way to tell whether the skill makes things worse.
The method, in short
What this does not tell you
  • One model, one moment. This is Opus 4.8 at one point in time. Other models, or a future version, may differ.
  • Autonomous first drafts. We measured what the model writes on its own, not a back-and-forth where you steer and correct it. In a real chat you would likely catch some of the fragile edge cases.
  • 24 jobs, not the whole world. The fields with only a couple of jobs are hints, not guarantees.
  • "No attack got through" means on this attack suite. Nothing we fired broke in. That is not a proof the code is safe.
  • A few runs failed for reasons unrelated to code quality. 2 of 480 had the model's reply leak into the saved file, and 1 was a test stub our recovery step grabbed by mistake; we count all of these as misses, which is the conservative call.
